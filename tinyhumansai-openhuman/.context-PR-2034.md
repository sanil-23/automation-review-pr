## PR Description
Harden `SecurityPolicy::is_path_allowed` against workspace symlink escapes. Resolves the existing path (or nearest existing parent) for workspace-relative paths before allowing them. Adds a Unix regression test covering both a symlink path and a not-yet-created child under a symlinked directory.

**Problem**: Issue #1927 reports that `is_path_allowed()` validates the path string but does not resolve symlinks. A path such as `escape_link` can look workspace-relative while resolving outside the workspace.

**Solution**: Add a helper that canonicalizes the path if it exists, or walks upward to canonicalize the nearest existing parent. In workspace-only mode, join relative paths to `workspace_dir` and reject them if the resolved existing prefix is outside the workspace.

## Linked Issues + Acceptance Criteria

### Issue #1927 — Security: Symlink bypass in is_path_allowed() — no canonicalization
- **Author**: @Liohtml
- **State**: OPEN
- **Core problem**: `is_path_allowed()` validates path string but doesn't resolve symlinks. A symlink inside workspace pointing to forbidden path (e.g., `/etc/shadow`) passes the check.
- **Impact**: Medium — agent can read/write forbidden files via symlink inside allowed workspace.
- **Suggested fix**: Merge `is_path_allowed()` and `is_resolved_path_allowed()` into single function that always canonicalizes, or enforce both are always called together.

**Acceptance criteria** (derived):
1. `is_path_allowed()` must resolve symlinks before checking workspace containment
2. Symlinks pointing outside workspace must be blocked
3. Existing callers that only use `is_path_allowed()` (without `is_resolved_path_allowed()`) must now be safe
4. Normal workspace-relative paths must still work

## PR-Issue Alignment (three-way verification)

**Issue asks**: Canonicalize in `is_path_allowed()` or enforce both checks are paired.
**PR claims**: Adds symlink resolution to `is_path_allowed()` via `resolve_existing_path_prefix` helper.
**Code does**: Adds the helper function, integrates it into `is_path_allowed()`, adds regression test.

**Alignment**: Good — PR addresses the core vulnerability. However:
- The issue suggested merging the two functions or enforcing pairing. The PR keeps both separate but adds defense-in-depth to `is_path_allowed()`. This is a valid alternative approach.
- **IMPORTANT**: CodeRabbit identified a critical gap — the `resolve_existing_path_prefix` function uses `current.exists()` which follows symlinks and returns `false` for **dangling** symlinks. This means a dangling symlink (pointing to a non-existent target) would bypass this new check, since `exists()` returns `false`, the loop climbs to the parent (which is inside workspace), and the path is allowed. This is a real escape vector for write operations (creating files through dangling symlinks pointing outside workspace).

**CRITICAL finding (actionable, NOT yet addressed):**
- `src/openhuman/security/policy.rs:386-395` — `resolve_existing_path_prefix` uses `current.exists()` which follows symlinks. Dangling symlinks (target doesn't exist) are treated as non-existent, causing the loop to climb to the workspace root and return `true`. This creates a write/create escape for dangling symlink targets. CodeRabbit suggests using `symlink_metadata()` to detect the symlink itself and `read_link()` to resolve the target. **This is a real security gap.**
