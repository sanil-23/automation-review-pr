## Quick Summary
*Generated: 2026-05-22T10:25:02.917Z*

**What it does** — Adds a new read-only RPC endpoint (`tool_registry.diagnostics`) that returns a summary of the tool inventory: how many tools exist, how many are enabled, a breakdown by transport type (MCP vs JSON-RPC), which tools look like they can write/mutate data (heuristic based on keywords like "create", "delete", "send"), and which tools are policy-related (security/approval endpoints). This is an observability feature so operators can audit what the AI agent can do without digging through logs.

**Safety & Breaking concerns** — **Low**

- **No security flaws**: The endpoint is read-only, takes no parameters, and returns only tool IDs and counts — no secrets, tokens, or PII. The `ToolPolicyDiagnostics` struct derives `Serialize` but not `Deserialize`, so it can't be used as an injection vector.
- **No breaking changes**: Purely additive — new RPC method, new type, new helper functions. Existing `list` and `get` endpoints are untouched (only the schema/controller count assertions in tests were bumped from 2→3).
- **Edge cases**: The `looks_write_capable` heuristic is substring-based and could false-positive on tool names that happen to contain "run" or "store" as substrings (e.g. a tool called `truncate` would match on nothing, but `instrument` would not match either — the delimiter checks `.`/`_` prevent most false positives). This is acceptable since it's labeled "possible" write surfaces, not authoritative.
- **Minor note**: The `diagnostics()` function calls `registry_entries()` which does a full tool scan including `block_in_place` for MCP client tools — same cost as `list_tools()`. Not a concern for a diagnostics endpoint but worth knowing.

**Bottom line** — Safe to merge: purely additive read-only diagnostics endpoint with good test coverage and no impact on existing behavior.
