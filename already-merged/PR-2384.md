## Quick Summary
*Generated: 2026-05-22T10:56:25.858Z*

**What it does** — When a user logs in via OAuth deep link (`openhuman://auth?token=...`) while connected to a remote (cloud-hosted) core, the app was sending the session token to a stale cached local core URL/token instead of the user's configured remote core. This PR clears the cached RPC URL and auth token right before storing the session, so the deep-link handler talks to the correct core. It also adds test coverage for the cache invalidation.

**Safety & Breaking concerns** — **Zero** breaking risk.

- The two `clear*` functions are well-established, side-effect-limited cache invalidators already used elsewhere. They reset in-memory variables and fire an invalidation event — no destructive behavior.
- The calls are placed at the top of `applySessionToken`, which is the right spot: cache is cleared *before* `storeSession` sends the RPC, ensuring the next resolution picks up the user's configured remote core.
- No new imports of sensitive APIs, no auth bypass vectors, no validation gaps.
- The change is additive (3 lines of production code, 11 lines of test) with zero deletions — existing behavior for local-core users is unaffected since clearing a cache that already points at the right place is a no-op in practice.
- Test properly mocks `coreRpcClient`, asserts both clears fire exactly once, and resets mocks between runs.

**Bottom line** — Safe to merge: a minimal, well-tested fix that solves a real user-facing bug (infinite onboarding loop on remote core) with no side effects.
