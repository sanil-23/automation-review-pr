## Quick Summary
*Generated: 2026-05-22T10:50:14.931Z*

**What it does** ‚Äî On Windows, when the app spawns shell/node/npm child processes, it strips the environment (`env_clear()`) and only passes a curated allowlist of variables. That allowlist was POSIX-only (HOME, TERM, SHELL, etc.), so on Windows the child process couldn't even locate executables ‚Äî `cmd.exe`, `PATH` extensions, temp dirs, and system root were all missing. This PR adds the essential Windows env vars (`SystemRoot`, `COMSPEC`, `PATHEXT`, `TEMP`, `USERPROFILE`, etc.) to the allowlist in all three tool modules, fixing "program not found" errors on Windows 11.

**Safety & Breaking concerns** ‚Äî **Zero** breaking risk.

- **Security**: The added variables are non-secret, read-only system paths and shell metadata (`SystemRoot`, `WINDIR`, `ProgramFiles`, `APPDATA`, etc.). They don't leak API keys, tokens, or user data. The allowlist approach is preserved ‚Äî only explicitly named vars pass through.
- **Breaking changes**: None. On non-Windows hosts these env vars simply won't exist, so the forwarding is a no-op. Existing POSIX behavior is untouched.
- **Edge cases**: The only nit is that the three `SAFE_ENV_VARS` lists are copy-pasted across `shell.rs`, `node_exec.rs`, and `npm_exec.rs` rather than shared via a constant ‚Äî but that's a pre-existing pattern, not a regression.
- **Tests**: Each module gets a regression test asserting the critical Windows vars are in the list. Tests are trivial and correct.

**Bottom line** ‚Äî Safe to merge: purely additive allowlist expansion for Windows compatibility with no effect on existing platforms, no secrets exposed, and the linked issue (#2379) is directly addressed.
