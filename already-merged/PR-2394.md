## Quick Summary
*Generated: 2026-05-22T11:26:38.740Z*

**What it does** — When a user clicks "Continue to chat" at the end of onboarding, four RPCs fire to finalize setup. Previously, if any RPC failed, the button silently died — the user was stuck with no feedback (#2081). This PR moves error handling from `ContextPage` (which swallowed it) down into `ContextGatheringStep` (which owns the button). Now the button shows a loading spinner while RPCs run, displays an error card if they fail, and reports to Sentry. The promise is also returned properly so callers can react to it.

**Safety & Breaking concerns** — **Low**.

- **No security issues.** No new inputs, no auth changes, no data exposure. Sentry capture uses the same tags as before.
- **No breaking changes.** The `onNext` contract changes from fire-and-forget to returning a promise, but the only caller is `ContextPage` which now does `return completeAndExit()` — and `ContextGatheringStep` wraps it in `Promise.resolve()` so both void and Promise returns work.
- **Double-click guard** is correct — `if (isCompleting) return` prevents re-entry, and `finally` clears it so retry is possible after failure.
- **Edge case: `setHasError(true)` + `setFinished(true)` on failure** — this reuses the existing error card UI path (the "your chat is ready" fallback), which is the right thing to do since it gives the user a way forward.
- **Test coverage** is solid: the old Sentry-in-ContextPage test is replaced by a Sentry-in-ContextGatheringStep test, plus loading/error state assertions.

**Bottom line** — Safe to merge: it's a focused UX fix that turns a silent dead button into proper loading + error feedback, with no behavioral regressions and good test coverage.
