## What it does

When the cloud classifier returns garbage (unparseable output) during background triage (e.g. Composio triggers), the second bad reply was treated as a fatal error. That bubbled up as an `Err` from `run_triage`, which made the Composio subscriber log a Sentry error event (`[composio][triage] run_triage failed` — issue #2322) that was noisy and non-actionable.

This PR changes the behavior so both cloud attempts (initial + retry) treat parse failures as retryable. After two bad cloud replies, the system falls through to the local classifier fallback, or returns `TriageOutcome::Deferred` if no local arm is configured — instead of crashing out with an error.

## Safety & Breaking concerns

**Breaking risk: Zero**

- **Security**: No new IO, no new dependencies, no auth changes. The diff is confined to error-classification logic inside `try_arm` in `evaluator.rs`. No injection surface, no data leak path.
- **Breaking changes**: The only behavioral change is that `CloudAfterRetry` parse failures now return `Retryable` instead of `Fatal`. This is strictly more lenient — existing fatal cases (auth errors, model-not-found, local-arm parse failures) are untouched. The match arm explicitly lists `LocalFallback` as the only fatal path.
- **Edge cases**: Both paths (local fallback available / not available) are covered by the two new regression tests. The counter assertions verify the exact dispatch sequence (cloud → cloud retry → local, or cloud → cloud retry → deferred).
- **Test caveat**: The author notes tests were blocked locally by a missing `libclang` on Windows — CI needs to confirm they actually pass. This is a CI validation gap, not a code concern.

## Bottom line

Safe to merge once CI confirms the two new tests pass — the change is a narrow, well-tested error-reclassification that eliminates noisy Sentry alerts without altering any success paths.
