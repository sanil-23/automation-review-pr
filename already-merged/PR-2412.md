## What it does

When a chat thread hits a transient error (rate limit, timeout, network failure, provider 503), the app was caching that thread's broken session state. On retry, it reused the poisoned session — so the thread stayed stuck showing "rate limited" even though a new thread worked fine.

This PR fixes it by **not caching the web-channel session after transient errors**, forcing the next message in that thread to rebuild from scratch. It also reorders the error classifier so a 429 with "insufficient balance" is correctly tagged as `budget_exhausted` (not a transient rate limit), and adds a new `network` error class for connectivity failures. German i18n strings for an unrelated MCP server settings page are bundled in.

## Safety & Breaking concerns

**Breaking risk: Low**

**Security:**
- No auth/input validation changes. Error messages are sanitized — raw transport URLs/details are stripped before reaching the user. Good.
- The `budget_exhausted` reorder is correct: checking budget markers *before* the generic 429 matcher prevents misclassification.

**Breaking changes:**
- The only behavioral change is: transient errors now clear the cached session. Non-transient errors (auth, budget, model, context) are explicitly excluded and keep existing behavior. This is well-scoped.
- The `chat_error` event shape is unchanged — same fields, same structure. Downstream consumers won't break.

**Edge cases:**
- The session is silently discarded (just a `log::debug!`). If the session held expensive state (long context window, tool state), that's rebuilt from transcript on retry — which is the intended behavior per the issue. No data loss.
- The `network` classifier uses broad substring matching (`"dns"`, `"connection reset"`). Could theoretically match a user message that somehow ends up in the error string, but `classify_inference_error` only receives provider error strings, not user input. Fine.
