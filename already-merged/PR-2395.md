## Quick Summary
*Generated: 2026-05-22T11:29:39.295Z*

**What it does** — When a Windows user tries to "Reset local data" and it fails because another OpenHuman window/process has the `.openhuman` directory locked, the app now shows a helpful message ("Close all OpenHuman windows and try again") instead of a raw OS error in Chinese/system locale. It detects Windows error codes 32 (sharing violation) and 33 (lock violation) and routes them through a shared formatter. Non-lock errors keep the existing generic message.

**Safety & Breaking concerns** — **Zero**

- No security issues: no new inputs accepted, no auth changes, no data exposure. The path/label in the error message were already being displayed.
- No breaking changes: the only behavioral difference is improved error text on Windows lock failures. Non-Windows platforms and non-lock errors produce the identical message as before.
- Edge cases are handled: `cfg!(windows)` guard means the lock-detection branch is dead code on macOS/Linux (always returns false). Tests cover both the raw OS error code matching and the full message formatting.
- The `is_windows_file_lock_raw_os_error` helper is tested independently from the `cfg!(windows)` gate, so code correctness is verified on all CI platforms even though the Windows-specific integration test only runs on Windows.

**Bottom line** — Safe to merge: a minimal, well-tested UX improvement for a Windows-only edge case with no risk to other platforms or existing behavior.
