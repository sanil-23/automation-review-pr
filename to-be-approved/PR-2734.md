### What it does

External tools like Claude Desktop, Cursor, or Zed can connect to OpenHuman as an MCP server — but until now they could only see and call *tools*. They couldn't pull OpenHuman's identity prompts (who it is, how it talks, what each sub-agent does) into their own conversations.

This PR adds two new MCP methods — `resources/list` and `resources/read` — that let those external clients discover and fetch 21 bundled markdown files: the 3 core identity files (`IDENTITY.md`, `SOUL.md`, `USER.md`) and the 18 built-in sub-agent prompts. The files are baked into the binary at compile time (`include_str!`), so there's no file I/O, no database calls, and no dynamic content at runtime.

---

### Safety & Breaking concerns

**Breaking risk: Zero**

The change is purely additive — two new JSON-RPC methods that only fire when a client explicitly calls them. Existing `tools/list` / `tools/call` paths are untouched. The `initialize` response gains a `resources` capability block, which is backwards-compatible per the MCP spec (clients that don't know about resources ignore it).

**Security**: Clean.
- No filesystem access — all content is `include_str!`-embedded at compile time
- No user-specific data exposed — these are the same prompt files already in the public repo
- No auth bypass — `SecurityPolicy` is unchanged; the resources are read-only
- Input validation present: missing `uri` → `-32602`, unknown URI → `-32002`
- No injection surface — URIs are matched against a static `const` array, not interpolated into queries

**Edge cases**: Well-handled.
- `cursor` param accepted but ignored (spec-compliance for single-page catalog)
- 11 unit tests cover: catalog parity vs `BUILTINS`, per-URI content distinctness (catches copy-paste bugs), required-field shape, both error paths, and protocol-level happy paths
- The catalog-parity test locks the subagent list — if someone adds a new subagent to `BUILTINS` without adding it here, CI fails

**Minor nit** (non-blocking): `read_resource_result` does a linear scan of ~21 entries. Fine at this scale, but worth noting if the catalog ever grows significantly.

---

### Bottom line

**Safe to merge** — this is a clean, additive, well-tested feature with no security concerns, no breaking changes, and full CI green.
