## Quick Summary
*Generated: 2026-05-22T08:30:47.215Z*

**What it does** — When a user's Discord bot token goes stale, clicking the Discord card on the Channels page used to log them out of the entire app. That happened because Discord's 401 error message contained the words "401" and "unauthorized," which tripped a global classifier that thought the *OpenHuman* session had expired. This PR rewrites Discord 401/403 error messages at the source (the Discord API client in Rust) so they no longer contain those trigger substrings. Instead, the user sees a helpful message like "bot token was rejected — go to Settings → Channels → Discord to fix it," and stays logged in.

**Safety & Breaking concerns** — **Low** risk.

- **No security issues.** The change actually *improves* security posture: upstream Discord error bodies (which could contain arbitrary text) are no longer spliced into user-facing messages. The raw body is still logged at `debug` level for triage.
- **No breaking changes to happy paths.** When the bot token is valid (200 responses), behavior is completely unchanged. Non-auth errors (400, 404, 5xx) keep the same verbose format they had before.
- **Error message wording changed for 401/403 only.** No external consumer pins the exact string. The PR includes a cross-module regression test (`is_session_expired_error_skips_discord_rewrap_for_2285`) that pins both the Discord wording and the classifier behavior, so future drift fails CI.
- **Edge case coverage is solid.** All six `anyhow::bail!` sites in the Discord API client are routed through the new helper. Tests cover 401, 403, and 5xx paths. The `is_session_expired_error` classifier was verified on the main branch — it already has HTTP-method-prefix gating (`msg.starts_with("GET /")` etc.), so even without this PR the classifier *should* skip Discord errors that start with "Discord." This PR is defense-in-depth, which is the right call since the classifier has proven fragile.
- **One minor nit**: the `let _ = body;` line in `format_discord_http_error` is a deliberate "I'm ignoring this on purpose" marker, not dead code — well-documented.

**Bottom line** — Safe to merge: a clean, narrowly-scoped defense-in-depth fix with thorough tests that prevents a real user-facing logout bug without touching any shared infrastructure.
